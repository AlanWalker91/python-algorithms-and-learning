## 1. 知识框架总览

```
第20章 并发执行器
│
├── Future 对象
│   ├── concurrent.futures.Future    # 线程/进程池的 Future
│   ├── asyncio.Future               # asyncio 的 Future
│   ├── 状态：pending → running → finished
│   ├── .done()     # 非阻塞，检查是否完成
│   ├── .result()   # 阻塞，获取结果（或重新抛出异常）
│   └── .add_done_callback()  # 完成时回调
│
├── Executor 接口（统一API）
│   ├── ThreadPoolExecutor
│   │   ├── max_workers 默认 = min(32, cpu_count + 4)
│   │   └── 适合：I/O 密集型任务
│   └── ProcessPoolExecutor
│       ├── max_workers 默认 = cpu_count()
│       └── 适合：CPU 密集型任务（绕开GIL）
│
├── 两种提交方式
│   ├── executor.map(fn, iterable)
│   │   ├── 按提交顺序返回结果
│   │   ├── 有异常时在迭代结果时抛出
│   │   └── 适合：同一函数 + 不同参数
│   └── executor.submit(fn, *args) → Future
│       ├── 配合 futures.as_completed()
│       ├── 按完成顺序产出 Future
│       └── 适合：不同函数 / 需要感知完成顺序
│
└── 使用模式
    ├── 上下文管理器（推荐）
    │   with executor as e: e.map(...) / e.submit(...)
    └── 错误处理：future.result() 重新抛出异常
```

## 2. Future 对象：异步计算的凭证

> **⚠️ 重点**：Future 对象代表"未来某时刻会完成的操作"，是 `concurrent.futures` 和 `asyncio` 的核心概念。**不应自己创建 Future，只能由框架创建**。

### 2.1 Future 的状态机

```
创建时 → pending（待定）
开始执行 → running（运行中）
执行完毕 → finished（完成）
  ├── 返回结果
  └── 抛出异常
```

### 2.2 两种 Future 的对比

| | `concurrent.futures.Future` | `asyncio.Future` |
|--|---|---|
| 用途 | 线程池/进程池 | asyncio 事件循环 |
| `.result()` 阻塞 | ✅ 会阻塞，支持 timeout | ❌ 不阻塞，需用 `await` |
| 获取结果 | `f.result()` | `await f` |
| 创建方式 | `executor.submit(fn)` | asyncio 内部创建 |

### 2.3 Future 的核心方法

```python
from concurrent import futures

# 假设通过 executor.submit 得到了 future
with futures.ThreadPoolExecutor(max_workers=3) as executor:
    future = executor.submit(some_function, arg1, arg2)

    # done()：非阻塞，检查是否完成
    print(future.done())     # False（可能还在运行）

    # result()：阻塞，直到有结果
    # 如果函数抛了异常，result() 会重新抛出
    result = future.result(timeout=5)  # 最多等5秒，超时抛 TimeoutError

    # add_done_callback()：完成时自动调用回调
    def on_done(f):
        print(f'完成，结果是：{f.result()}')
    future.add_done_callback(on_done)
    # 注意：回调在同一个工作线程/进程中运行，不是主线程
```

## 3. ThreadPoolExecutor：线程池执行器

### 3.1 最简用法：executor.map

```python
from concurrent import futures
from flags import save_flag, get_flag, main  # 从示例模块导入


def download_one(cc: str) -> str:
    """下载单个国旗图像（在工作线程中执行）。"""
    image = get_flag(cc)
    save_flag(image, f'{cc}.gif')
    print(cc, end=' ', flush=True)
    return cc


def download_many(cc_list: list[str]) -> int:
    """使用线程池并发下载所有国旗。

    ThreadPoolExecutor 作为上下文管理器使用：
    - 退出时自动调用 executor.shutdown(wait=True)
    - 确保所有线程完成后才继续执行
    """
    with futures.ThreadPoolExecutor() as executor:
        # executor.map 与内置 map 类似，但 download_one 在多个线程中并发调用
        # 返回一个生成器，产出各函数调用的返回值（按提交顺序）
        res = executor.map(download_one, sorted(cc_list))
    return len(list(res))       # 如果有线程抛出异常，此处会重新抛出


if __name__ == '__main__':
    main(download_many)
```

### 3.2 max_workers 的计算逻辑

```python
# ThreadPoolExecutor 的默认 max_workers（Python 3.8+）：
import os
max_workers = min(32, os.cpu_count() + 4)

# 为什么这样设计？
# - 至少5个职程（os.cpu_count() 通常 >= 1，所以 +4 保证至少有5个）
# - 对释放了GIL的CPU密集型任务，最多32个核
# - 避免在多核设备上不知不觉使用过多资源

# 实际建议：
# - I/O 密集型：可以设置更大，如 max_workers=100（线程等待时不占CPU）
# - CPU 密集型：用 ProcessPoolExecutor，max_workers=cpu_count()
```

### 3.3 使用 submit + as_completed 观察 Future

```python
def download_many(cc_list: list[str]) -> int:
    """使用 submit + as_completed，可以观察每个 Future 的状态。"""
    cc_list = cc_list[:5]      # 演示用，只下载5个

    with futures.ThreadPoolExecutor(max_workers=3) as executor:
        to_do = []
        # 提交任务，每次提交返回一个 Future 对象
        for cc in sorted(cc_list):
            future = executor.submit(download_one, cc)
            to_do.append(future)
            print(f'Scheduled for {cc}: {future}')
            # 输出如：<Future at 0x... state=running>
            #         <Future at 0x... state=pending>

        # as_completed：产出已完成的 Future（按完成顺序，不是提交顺序）
        for count, future in enumerate(futures.as_completed(to_do), 1):
            res: str = future.result()      # 此时 future 已完成，不会阻塞
            print(f'{future} result: {res!r}')

    return count
```

## 4. ProcessPoolExecutor：进程池执行器

### 4.1 从线程池切换到进程池（只需改一行）

```python
# 线程池（I/O密集型）
def download_many(cc_list):
    with futures.ThreadPoolExecutor() as executor:       # ← 线程
        res = executor.map(download_one, sorted(cc_list))
    return len(list(res))

# 进程池（CPU密集型）—— 只需把 ThreadPoolExecutor 换成 ProcessPoolExecutor
def compute_many(numbers):
    with futures.ProcessPoolExecutor() as executor:      # ← 进程
        res = executor.map(is_prime, numbers)
    return list(res)

# 两个类都实现了相同的 Executor 接口，API 完全一致
```

### 4.2 使用 ProcessPoolExecutor 重写素数检测

```python
import sys
from concurrent import futures
from time import perf_counter
from typing import NamedTuple
from primes import is_prime, NUMBERS


class PrimeResult(NamedTuple):
    n: int
    flag: bool
    elapsed: float


def check(n: int) -> PrimeResult:
    """检测一个数是否为素数，计时并返回结果。"""
    t0 = perf_counter()
    res = is_prime(n)
    return PrimeResult(n, res, perf_counter() - t0)


def main() -> None:
    # 未提供命令行参数时，让 ProcessPoolExecutor 自动决定进程数
    workers = None if len(sys.argv) < 2 else int(sys.argv[1])

    # 在 with 块外创建，以便提前获取 _max_workers 属性显示
    executor = futures.ProcessPoolExecutor(workers)
    actual_workers = executor._max_workers  # type: ignore

    print(f'Checking {len(NUMBERS)} numbers with {actual_workers} processes:')
    t0 = perf_counter()

    # 倒序提交（让最大的数先检测，充分利用并行）
    numbers = sorted(NUMBERS, reverse=True)

    with executor:
        # executor.map 返回结果的顺序与 numbers 的顺序一致
        # ⚠️ 这意味着：最难计算的数排第一，其他结果要等它完成后才输出
        for n, prime, elapsed in executor.map(check, numbers):
            label = 'P' if prime else ' '
            print(f'{n:16} {label} {elapsed:9.6f}s')

    time = perf_counter() - t0
    print(f'Total time: {time:.2f}s')


if __name__ == '__main__':
    main()
```

**⚠️ 注意 executor.map 的顺序保证**：

```python
# executor.map 保证结果顺序与提交顺序一致
# 后果：如果第一个任务耗时很长，其他结果都要等它

# 示例：倒序提交 20 个数
# 9999999999999917（最大素数）排第一，耗时 9.5 秒
# 程序先快速完成其他数的检测，然后卡住等待第一个结果
# → 所有结果在第一个完成后瞬间全部输出

# 解决方案：如果需要"先完成先输出"，用 submit + as_completed
with executor:
    to_do = {executor.submit(check, n): n for n in numbers}
    for future in futures.as_completed(to_do):  # 按完成顺序产出
        n, prime, elapsed = future.result()
        label = 'P' if prime else ' '
        print(f'{n:16} {label} {elapsed:9.6f}s')
```

## 5. Executor.map vs submit + as_completed

### 5.1 Executor.map 演示

```python
from time import sleep, strftime
from concurrent import futures


def display(*args):
    """打印带时间戳的消息。"""
    print(strftime('[%H:%M:%S]'), end=' ')
    print(*args)


def loiter(n):
    """休眠 n 秒，模拟耗时操作，返回 n * 10。"""
    display(f'loiter({n}): doing nothing for {n}s...')
    sleep(n)
    display(f'loiter({n}): done.')
    return n * 10


def main():
    display('Script starting.')
    executor = futures.ThreadPoolExecutor(max_workers=3)

    # executor.map 是非阻塞的：立即返回一个生成器
    # 任务开始排队执行，但 results 变量此时只是一个生成器对象
    results = executor.map(loiter, range(5))
    display('results:', results)               # 立即打印，不等待任何任务

    display('Waiting for individual results:')
    for i, result in enumerate(results):
        # 迭代时才阻塞：等待第 i 个任务完成（按提交顺序）
        # loiter(0) 已完成（sleep 0秒），立即返回
        # loiter(1) 需要等 1 秒才返回
        # loiter(2) 需要等 2 秒
        display(f'result {i}: {result}')


if __name__ == '__main__':
    main()

# 关键观察：
# - max_workers=3，所以 loiter(0/1/2) 立即开始，loiter(3/4) 等待
# - 迭代 results 时，按提交顺序等待，不是按完成顺序
# - loiter(0) 最先完成（sleep 0），主线程立即得到 result 0
# - 然后等 loiter(1) 完成，得到 result 1...以此类推
```

### 5.2 两种方式的适用场景对比

| | `executor.map` | `executor.submit + as_completed` |
|--|----------------|----------------------------------|
| **结果顺序** | 按提交顺序（同步）| 按完成顺序（异步）|
| **函数** | 同一个函数+不同参数 | 可以是不同函数 |
| **执行器** | 一个执行器 | 可以来自多个执行器 |
| **错误处理** | 迭代时才抛出 | `future.result()` 时抛出 |
| **代码简洁性** | 极简（一行）| 稍复杂（需要循环）|
| **适合场景** | 批量同构任务 | 需要感知完成顺序的任务 |

```python
# 示例：submit + as_completed 的灵活性

# 可以混合不同类型的任务
with futures.ThreadPoolExecutor() as executor:
    future_to_task = {}

    # 提交不同的函数
    future_to_task[executor.submit(download_image, url)] = ('image', url)
    future_to_task[executor.submit(fetch_metadata, url)] = ('meta', url)
    future_to_task[executor.submit(check_links, url)] = ('links', url)

    # 按完成顺序处理（先完成的先处理）
    for future in futures.as_completed(future_to_task):
        task_type, task_url = future_to_task[future]   # 通过字典找回上下文
        try:
            result = future.result()
            print(f'{task_type} for {task_url}: done')
        except Exception as exc:
            print(f'{task_type} for {task_url} failed: {exc}')
```

## 6. 完整示例：带进度条和错误处理的下载器

> 这是本章最完整的示例，展示了生产级并发代码的实现模式。

### 6.1 基础下载函数（顺序版）

```python
from collections import Counter
from http import HTTPStatus
import httpx
import tqdm
from flags2_common import main, save_flag, DownloadStatus

DEFAULT_CONCUR_REQ = 1
MAX_CONCUR_REQ = 1


def get_flag(base_url: str, cc: str) -> bytes:
    """下载国旗图像，HTTP错误时抛出异常。"""
    url = f'{base_url}/{cc}/{cc}.gif'.lower()
    resp = httpx.get(url, timeout=3.1, follow_redirects=True)
    resp.raise_for_status()  # 非2xx状态码时抛出 HTTPStatusError
    return resp.content


def download_one(cc: str, base_url: str, verbose: bool = False) -> DownloadStatus:
    """下载单个国旗，处理404错误。

    其他错误向上冒泡，由 download_many 统一处理。
    """
    try:
        image = get_flag(base_url, cc)
    except httpx.HTTPStatusError as exc:
        res = exc.response
        if res.status_code == HTTPStatus.NOT_FOUND:
            # 404：标记为"未找到"，不作为错误
            status = DownloadStatus.NOT_FOUND
            msg = f'not found: {res.url}'
        else:
            raise                              # 其他HTTP错误：重新抛出
    else:
        save_flag(image, f'{cc}.gif')
        status = DownloadStatus.OK
        msg = 'OK'

    if verbose:
        print(cc, msg)
    return status
```

### 6.2 并发下载（线程池版）⭐

```python
from collections import Counter
from concurrent.futures import ThreadPoolExecutor, as_completed
import httpx
import tqdm
from flags2_common import main, DownloadStatus
from flags2_sequential import download_one

DEFAULT_CONCUR_REQ = 30     # 默认30个并发连接
MAX_CONCUR_REQ = 1000       # 最大并发数上限（安全限制）


def download_many(
    cc_list: list[str],
    base_url: str,
    verbose: bool,
    concur_req: int
) -> Counter[DownloadStatus]:
    """并发下载所有国旗，显示进度条，统计结果。"""

    counter: Counter[DownloadStatus] = Counter()

    # max_workers = concur_req（由命令行参数或默认值决定）
    with ThreadPoolExecutor(max_workers=concur_req) as executor:
        # ⭐ 核心技巧：把 future 对象映射到对应的上下文（国家代码）
        # 这样在异常处理时可以知道是哪个任务出了问题
        to_do_map: dict[futures.Future, str] = {}

        for cc in sorted(cc_list):
            # submit 提交任务，立即返回 Future 对象（非阻塞）
            future = executor.submit(download_one, cc, base_url, verbose)
            to_do_map[future] = cc             # Future → 国家代码

        # as_completed 产出已完成的 Future（按完成顺序）
        done_iter = as_completed(to_do_map)

        # tqdm 包装迭代器，显示进度条
        if not verbose:
            done_iter = tqdm.tqdm(done_iter, total=len(cc_list))

        for future in done_iter:
            try:
                # future.result()：获取结果，或重新抛出异常
                # 因为是 as_completed 产出的，这里不会阻塞
                status = future.result()
            except httpx.HTTPStatusError as exc:
                error_msg = 'HTTP error {resp.status_code} {resp.reason_phrase}'
                error_msg = error_msg.format(resp=exc.response)
            except httpx.RequestError as exc:
                error_msg = f'{exc} {type(exc)}'.strip()
            except KeyboardInterrupt:
                break                          # Ctrl+C 优雅退出
            else:
                error_msg = ''                 # 没有异常

            if error_msg:
                status = DownloadStatus.ERROR
                if verbose:
                    # 从字典中找回对应的国家代码（因为迭代的是future，不是cc）
                    cc = to_do_map[future]
                    print(f'{cc} error: {error_msg}')

            counter[status] += 1

    return counter


if __name__ == '__main__':
    main(download_many, DEFAULT_CONCUR_REQ, MAX_CONCUR_REQ)
```

### 6.3 惯用模式：future → 上下文 的字典

```python
# ⭐ 这是使用 as_completed 时最重要的模式：
# 把 Future 映射到相关上下文，以便在完成时找回信息

# 方式一：Future → 输入数据
future_to_url = {executor.submit(download, url): url for url in urls}
for future in as_completed(future_to_url):
    url = future_to_url[future]       # 找回对应的URL
    try:
        data = future.result()
    except Exception as exc:
        print(f'{url} raised {exc}')

# 方式二：Future → 任务描述
future_to_task = {}
future_to_task[executor.submit(fn1, arg1)] = 'task1 description'
future_to_task[executor.submit(fn2, arg2)] = 'task2 description'
```

## 7. 性能对比：顺序 vs 线程 vs 协程

> 下载20个国旗图像的实测数据：

```
$ python3 flags.py
20 flags downloaded in 7.26s    ← 顺序下载

$ python3 flags_threadpool.py
20 flags downloaded in 1.37s    ← 线程池（约快5倍）

$ python3 flags_asyncio.py
20 flags downloaded in 1.36s    ← asyncio（与线程池相近）
```

**为什么线程和协程速度相近？**

```
对于 I/O 密集型任务（网络下载）：
- 大多数时间消耗在等待网络响应
- 线程和协程都能在等待期间处理其他任务
- 性能差异主要取决于网络延迟，不是并发模型

线程 vs 协程 的实质区别在于：
- 协程的内存占用远低于线程（数千字节 vs 数百KB）
- 协程没有上下文切换开销
- 协程更适合超高并发（数万个并发连接）
- 线程更容易阻塞（任何阻塞调用都会影响整个线程）
```

## 8. 最佳实践与陷阱

### ✅ 最佳实践

```python
# 1. 用上下文管理器，确保执行器正确关闭
with futures.ThreadPoolExecutor(max_workers=10) as executor:
    results = executor.map(fn, args)
# with 块结束时自动等待所有任务完成

# 2. I/O密集型用线程，CPU密集型用进程
io_executor = futures.ThreadPoolExecutor()      # I/O密集
cpu_executor = futures.ProcessPoolExecutor()    # CPU密集

# 3. 用 future_to_context 字典保存上下文
to_do_map = {executor.submit(fn, arg): arg for arg in args}
for future in as_completed(to_do_map):
    original_arg = to_do_map[future]  # 完成时找回输入数据

# 4. 总是处理异常
for future in as_completed(futures_list):
    try:
        result = future.result()
    except Exception as exc:
        print(f'Task failed: {exc}')

# 5. 用 tqdm 显示进度条（用户体验）
from tqdm import tqdm
done_iter = tqdm(as_completed(futures_list), total=len(futures_list))
```

### ❌ 常见陷阱

```python
# ❌ 错误1：手动创建 Future 对象
f = futures.Future()    # 不要这样做！Future 由框架创建

# ❌ 错误2：在 executor.map 返回值上直接调用 len
res = executor.map(fn, args)
print(len(res))         # 错误！map 返回生成器，没有 len
print(len(list(res)))   # 正确（但会消耗生成器）

# ❌ 错误3：在调用 as_completed 之前使用 future.result()
futures_list = [executor.submit(fn, arg) for arg in args]
for f in futures_list:
    f.result()          # 如果第一个任务还未完成，会阻塞！
                        # 正确方式：用 as_completed 再调用 result()

# ❌ 错误4：忘记 ProcessPoolExecutor 的序列化要求
def my_func(data):
    return data['key']  # data 必须可以被 pickle 序列化

with futures.ProcessPoolExecutor() as executor:
    # 如果 args 中有 lambda 或未序列化对象，会在这里失败
    results = executor.map(my_func, args)

# ❌ 错误5：对 I/O 密集型任务使用 ProcessPoolExecutor
with futures.ProcessPoolExecutor() as executor:    # ← 进程启动开销高
    results = executor.map(download_flag, urls)    # ← 对I/O任务没有额外收益

# ✅ 正确：I/O 任务用线程
with futures.ThreadPoolExecutor() as executor:
    results = executor.map(download_flag, urls)

# ❌ 错误6：在多线程程序中直接更新共享列表（非线程安全）
results = []
def worker(n):
    results.append(n * 2)   # ← 多线程并发写入，可能导致数据竞争

# ✅ 正确：用队列或直接通过 executor.map 收集结果
results = list(executor.map(lambda n: n * 2, numbers))
```

### 何时选择哪种执行器

```
任务特征                    →  推荐方案
─────────────────────────────────────────────────────
网络请求（HTTP/API调用）    →  ThreadPoolExecutor 或 asyncio
磁盘I/O（文件读写）         →  ThreadPoolExecutor
数据库查询                  →  ThreadPoolExecutor 或 asyncio
数值计算（NumPy/scikit）    →  ThreadPoolExecutor（C扩展释放GIL）
纯Python CPU密集型计算      →  ProcessPoolExecutor
需要超高并发（>1000连接）   →  asyncio（第21章）
简单批量任务，不关心顺序    →  executor.map
需要感知完成顺序            →  executor.submit + as_completed
不同类型的混合任务          →  executor.submit + as_completed
```
