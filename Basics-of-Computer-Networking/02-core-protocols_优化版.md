# 一、TCP 协议：网络世界的"签收快递"

## 1.1 TCP 到底在解决什么问题？

在阶段 1 我们知道，网络层的 IP 协议负责把数据包从 A 送到 B。但 IP 协议有一个很大的问题：

**它是"尽力而为"的——不保证数据一定能到，不保证顺序，不保证不重复。**

这就像你把信扔进邮筒：邮局会尽力送，但信可能丢了、可能顺序乱了、可能被送了两份。

对于打电话、看视频，丢几个包问题不大。但对于转账、下载文件、发邮件——**你能接受数据丢一部分吗？不能。**

所以 TCP 就是在这个"不可靠"的 IP 层之上，构建出一个"可靠"的传输通道。

> **一句话定义 TCP**：TCP 是一个面向连接的、可靠的、基于字节流的传输层协议。
> 
> 面向连接 → 先"打电话"建立连接，再传数据
> 可靠 → 保证数据不丢、不乱、不重复
> 字节流 → 数据像水流一样没有边界（这会导致粘包问题，后面讲）

## 1.2 三次握手：为什么要"先打电话"？

### 先理解为什么需要握手

你和朋友打电话：

```
你：  "喂，听得到吗？"
朋友："听得到，你能听到我吗？"
你：  "能听到，咱们开始说吧。"
```

三句话，缺一不可。为什么？

- 如果只有第 1 句：你说了话，但不知道朋友有没有听到
- 如果只有前 2 句：朋友说了话，但不确定你能不能听到他的声音
- 第 3 句之后：**双方都确认了对方的"听"和"说"能力**

TCP 三次握手的逻辑完全一样——**用最少的交互次数，确认双方的收发能力。**

### 具体过程

```
    你的电脑 (Client)                     百度服务器 (Server)
    状态: CLOSED                          状态: LISTEN（在等电话）
        │                                      │
   ①    │────── SYN, seq=1000 ────────────────>│
        │    "喂，我想和你建立连接"              │
        │    "我的初始序列号是 1000"             │
        │                                      │
        │  你→ SYN_SENT                   它→ SYN_RCVD
        │                                      │
   ②    │<───── SYN+ACK, seq=2000, ack=1001 ──│
        │    "好的，我也想和你连接"              │
        │    "我的初始序列号是 2000"             │
        │    "我期望你下次发 1001"               │
        │                                      │
   ③    │────── ACK, ack=2001 ────────────────>│
        │    "收到，开始传数据吧"                │
        │                                      │
        │  你→ ESTABLISHED              它→ ESTABLISHED
```

### 几个关键细节（面试追问点）

**seq（序列号）是什么？为什么不从 0 开始？**

每个 TCP 连接的初始序列号是**随机生成**的。这是为了安全：如果总是从 0 开始，攻击者很容易伪造报文。随机序列号让伪造变得困难。

**ack（确认号）是什么意思？**

ack=1001 的意思是"我已经收到了 1000 及以前的所有数据，下次请从 1001 开始发"。这是**累积确认**机制。

**为什么不是两次握手？**

这是面试最高频的追问。很多人会说"为了确认双方收发能力"，这没错但不够深入。更本质的原因是：

> **防止历史的、失效的连接请求突然到达服务端，导致资源浪费。**
> 
> 想象这个场景：你之前发了一个连接请求 SYN，但网络很慢，这个 SYN 在网络里晃悠了很久。
> 后来你不等了，重新建了一个连接（已经正常通信完了）。
> 突然，那个旧的 SYN 终于到了服务端。
> 
> 如果只有两次握手：服务端收到旧 SYN 后直接分配资源、进入连接状态、开始等数据。
> 但你根本不会发数据过来——服务端就白白浪费了资源。
> 
> 有了第三次握手：服务端回了 SYN+ACK 后，需要等你的 ACK 才进入连接状态。
> 你收到这个过期的 SYN+ACK 后会发现"这不是我发起的"，直接发 RST 拒绝。
> 服务端收到 RST，释放资源。问题解决。

### 🔧 动手验证：用 Python 观察握手

```python

运行这个脚本后，用 Wireshark 过滤 tcp.port == 80 
你能看到真实的 SYN → SYN+ACK → ACK
"""

sock.connect(('httpbin.org', 80))  # ← 这一行就触发了三次握手
print("连接成功！握手完成。")

# 关闭连接（会触发四次挥手）

```

看到了吗？你在理论中学到的 SYN、SYN+ACK、ACK，现在在 Wireshark 里有了真实的样子。

## 1.3 四次挥手：为什么分手比握手多一步？

### 直觉理解

握手时，服务端的"同意连接"（SYN）和"确认收到"（ACK）可以合并成一个包发。

但分手时不行，原因很简单：

> **收到对方的"我要断开"请求时，你可能还有数据没说完。**

就像打电话：
```
你：  "我说完了，要挂了。"        ← FIN
朋友："好的我知道了..."           ← ACK
朋友："...但我还有几句话要说..."  ← （继续发数据）
朋友："好了我也说完了，挂吧。"    ← FIN
你：  "好的，挂了。"              ← ACK
```

"知道了"和"我也说完了"这两步**不能合并**，因为中间可能还有数据要发。所以是四次。

### 具体过程

```
    你的电脑 (Client)                     服务器 (Server)
    状态: ESTABLISHED                     状态: ESTABLISHED
        │                                      │
   ①    │────── FIN, seq=u ───────────────────>│  "我不发了"
        │  你→ FIN_WAIT_1                它→ CLOSE_WAIT
        │                                      │
   ②    │<───── ACK, ack=u+1 ─────────────────│  "知道了"
        │  你→ FIN_WAIT_2                      │
        │                                      │  （服务端可能还在发数据...）
        │                                      │
   ③    │<───── FIN, seq=w ───────────────────│  "我也不发了"
        │  你→ TIME_WAIT                 它→ LAST_ACK
        │                                      │
   ④    │────── ACK, ack=w+1 ─────────────────>│  "好的，再见"
        │                                它→ CLOSED
        │                                      │
        │  等待 2MSL（约 60 秒）...              │
        │  你→ CLOSED                          │
```

### TIME_WAIT：为什么要等 2MSL？

MSL = Maximum Segment Lifetime，一个报文在网络中能存活的最长时间（通常 30 秒）。

**等 2MSL 有两个原因**：

**原因 1：确保最后一个 ACK 能到达。**

如果你发的最后一个 ACK 丢了，服务端收不到，就会重发 FIN。
你在 TIME_WAIT 期间还能收到这个重发的 FIN 并重新回 ACK。
如果你不等直接关闭了，服务端重发的 FIN 就没人处理了。

**原因 2：等这个连接的所有报文从网络中消失。**

假设你马上用相同的 IP+端口 建了一个新连接，而旧连接的某个延迟报文突然到了——
新连接会把旧数据当成自己的数据，就乱了。等 2MSL 确保旧报文全部消亡。

### TIME_WAIT 太多怎么办？（面试高频）

这是一个非常实际的问题，特别是在**高并发短连接**场景（比如压测时）：

```
问题：
  每个 TIME_WAIT 占一个端口，持续 60 秒
  高并发时每秒创建上千个连接 → 端口可能耗尽
  表现：Connection Refused 或无法建立新连接

排查方法：
  ss -tan | grep TIME-WAIT | wc -l    # 看有多少 TIME_WAIT

解决方案（面试时从最佳到兜底）：
  ① 使用长连接（keep-alive / 连接池）   ← 根本解决方案，减少连接创建
  ② tcp_tw_reuse = 1（允许复用 TIME_WAIT 端口）
  ③ 增大端口范围（ip_local_port_range）
  ④ 缩短 TIME_WAIT 时间（不太推荐，可能导致旧报文干扰）

测试开发视角：
  压测前一定要检查 TIME_WAIT 数量
  用连接池（如 requests.Session()）而不是每次创建新连接
```

### CLOSE_WAIT 太多怎么办？

这个比 TIME_WAIT 更严重，因为它几乎 100% 是代码 Bug：

```
CLOSE_WAIT 含义：
  对方已经发了 FIN（"我不发了"），你也回了 ACK（"知道了"），
  但你的代码没有调用 close()（"我也不发了"这一步没做）。

根因：代码中连接没有被正确关闭。常见于：
  - try 中出异常，但 finally 没有 close()
  - 用了连接池但没有归还连接
  - 忘记关闭 socket / 数据库连接 / HTTP 连接

排查：
  ss -tanp | grep CLOSE-WAIT    # 看是哪个进程积压了 CLOSE_WAIT
  然后去检查那个程序的连接关闭逻辑
```

## 1.4 TCP 可靠传输的六大机制

面试中"TCP 如何保证可靠传输"是必问题。很多人只能说出一两个，你需要说全六个：

### 机制 1：序列号（保证有序）

每个字节都有唯一的编号。接收方根据编号重新排列，保证数据有序。

*为什么需要？——网络中不同的包可能走不同的路，到达顺序可能乱。*

### 机制 2：确认应答 ACK（保证送达）

接收方每收到数据就回复 ACK："我收到了 xxx 之前的所有字节。"

*为什么用累积确认？—— ACK=1000 表示 999 及之前都收到了，比逐个确认效率高。*

### 机制 3：超时重传（丢包保险）

发送后启动定时器，超时没收到 ACK 就重传。超时时间（RTO）根据网络延迟（RTT）动态计算。

*为什么是动态的？—— 网络延迟不是固定的，WiFi 和光纤差别很大。*

### 机制 4：滑动窗口（流量控制）

接收方通过 window size 告诉发送方"我的缓冲区还能装多少数据"，发送方不能发超过这个量的数据。

*解决什么问题？—— 防止发送方发太快，接收方处理不过来。*

```
想象一个水管：
  发送方是水龙头，接收方是水桶。
  窗口大小 = 水桶的剩余容量。
  如果水桶满了（窗口=0），水龙头就得停。
```

### 机制 5：拥塞控制（不压垮网络）

滑动窗口管的是"不压垮接收方"，拥塞控制管的是"不压垮整个网络"。

四个算法（面试常考，但不需要记太细，理解思想就行）：

```
核心思想："试探性地加速，一旦出问题立刻减速"

① 慢启动：从小窗口开始，指数增长（1→2→4→8→16...）
   为什么叫"慢"？因为起步是 1，比直接开满窗口"慢"
   实际增长速度很快（指数级）

② 拥塞避免：窗口到达阈值后，线性增长（16→17→18→19...）
   为什么要减速？因为快到网络承受极限了，要小心探测

③ 快重传：收到 3 个重复 ACK → 立即重传，不等超时
   为什么 3 个？1 个可能是正常乱序，3 个基本确定是丢了

④ 快恢复：发生快重传后，不回到慢启动
   窗口减半 → 继续线性增长
   为什么不回到 1？因为能收到重复 ACK 说明网络没有完全瘫痪
```

### 机制 6：校验和（保证完整）

发送方计算数据的校验和写入头部，接收方重新计算并对比。不一致则丢弃该报文。

*只是最后一道防线，主要靠前 5 个机制保证可靠。*

```
Wireshark 过滤器（这些能帮你在抓包中发现问题）：

  tcp.analysis.retransmission     → 看到"重传"
  tcp.analysis.duplicate_ack      → 看到"重复 ACK"（快重传的触发条件）
  tcp.analysis.zero_window        → 看到"窗口为零"（流量控制触发了）
  tcp.analysis.window_full        → 看到"窗口满"
```

## 1.5 TCP 粘包问题

### 为什么会有粘包？

TCP 是**字节流**协议——数据像水一样流过去，没有边界标记。

```
你连续发送了三条消息：
  发送：  "Hello"  "World"  "!"

接收方可能收到的情况（取决于 TCP 缓冲区和读取时机）：

  情况 1（正常）：  "Hello"  "World"  "!"     三次 recv
  情况 2（粘包）：  "HelloWorld!"              一次 recv 收到全部
  情况 3（拆包）：  "Hel"  "loWor"  "ld!"     被拆成了奇怪的三段
```

*为什么 UDP 没有粘包？—— 因为 UDP 是面向"数据报"的，每个报文有独立边界。*

### 解决方案

三种方案，从简单到推荐：

```
方案 1：固定长度
  每条消息固定 100 字节，不足补空格
  简单但浪费带宽

方案 2：分隔符
  每条消息以 \r\n 结尾
  适合文本协议，但如果消息内容本身包含 \r\n 就出问题

方案 3：长度前缀（最常用、最推荐）
  每条消息前面加 4 个字节，表示后面消息体的长度
  接收方先读 4 字节拿到长度，再精确读取对应字节数
  
  消息格式：[4字节长度][消息体]
  例如：    [0x00000005][Hello]     (5 字节的消息)
           [0x00000005][World]     (5 字节的消息)
```

### 🔧 动手验证：实现长度前缀协议

```python
"""自己实现一个带长度前缀的消息协议，亲手解决粘包"""
import struct
import socket

def send_msg(sock, text):
    """发送一条消息：4字节长度头 + 消息体"""
    data = text.encode('utf-8')
    header = struct.pack('>I', len(data))  # 大端序 4 字节无符号整数（> 表示大端序（网络传输的标准字节序），I 表示 4 字节无符号整数。作用：把一个整数变成固定 4 字节的二进制“头”。）
    sock.sendall(header + data)

def recv_msg(sock):
    """接收一条消息：先读 4 字节长度，再读对应长度的消息体"""
    # 精确读取 4 字节
    header = b''
    while len(header) < 4:
        chunk = sock.recv(4 - len(header))
        if not chunk:
            return None
        header += chunk
    
    # 解析长度
    msg_len = struct.unpack('>I', header)[0]
    
    # 精确读取消息体
    body = b''
    while len(body) < msg_len:
        chunk = sock.recv(msg_len - len(body))
        if not chunk:
            return None
        body += chunk
    
    return body.decode('utf-8')

# 试一试：在你的 TCP 服务器/客户端中用这两个函数代替 send/recv
# 你会发现不管发多快、发多少条，每条消息都能被完整接收
```

# 二、UDP：为什么有时候"不可靠"反而更好？

## 2.1 理解 UDP 的设计哲学

学完 TCP 你可能会想：既然 TCP 这么可靠，为什么还需要 UDP？

因为**可靠是有代价的**——握手要时间、重传要等待、流量控制要减速。

在某些场景下，**速度比完整性更重要**：
- 视频直播：丢几帧无所谓，但卡顿不可接受
- DNS 查询：只有一个小小的请求和响应，建连接太浪费
- 在线游戏：延迟 100ms 比丢一个位置更新包更致命

所以 UDP 的设计哲学是：**什么都不管，你发我就发，快就完事了。**

```
TCP 像打电话：先拨号、等对方接听、确认双方能说话，然后聊
UDP 像扔纸条：写完就扔过去，至于对方接到没有，不管

UDP 头部只有 8 字节（源端口 + 目的端口 + 长度 + 校验和）
TCP 头部最少 20 字节
```

## 2.2 TCP vs UDP：一张表搞定（面试必背）

| 维度 | TCP | UDP | 记忆方法 |
|------|-----|-----|----------|
| 连接 | 需要三次握手 | 无连接，直接发 | 电话 vs 扔纸条 |
| 可靠 | 保证不丢不乱 | 不保证 | 签收快递 vs 扔门口 |
| 顺序 | 保证有序 | 不保证 | 有编号 vs 无编号 |
| 传输方式 | 字节流（无边界） | 数据报（有边界） | 水流 vs 一封封信 |
| 头部大小 | 20 字节起 | 8 字节 | 重量级 vs 轻量级 |
| 速度 | 较慢（开销大） | 快（开销小） | — |
| 拥塞控制 | 有 | 无 | TCP 替网络着想 |
| 粘包 | 有 | 无 | 字节流的代价 |

## 2.3 面试加分回答

> "TCP 和 UDP 的选择本质上是**可靠性和实时性的权衡**。现代应用的一个趋势是在 UDP 上自己实现需要的可靠性——比如 HTTP/3 使用的 QUIC 协议，底层就是 UDP，但在应用层自己实现了连接管理、可靠传输和拥塞控制。这样既保留了 UDP 的灵活性，又能精确控制可靠性级别。"

# 三、HTTP/HTTPS：测试开发的"母语"

## 3.1 为什么说 HTTP 是测试开发的"母语"？

作为测试开发，你每天打交道最多的就是 HTTP：
- 写接口自动化测试 → 发 HTTP 请求、校验响应
- 调试接口问题 → 看 HTTP 报文
- 做性能测试 → 压 HTTP 接口
- 排查线上问题 → 看 HTTP 状态码和响应内容

所以 HTTP 你不能只是"知道"，你得**精通**。

## 3.2 HTTP 请求的完整解剖

一个 HTTP 请求由四部分组成，我们一个一个看：

```
POST /api/users HTTP/1.1              ← ① 请求行
Host: api.example.com                 ← ② 请求头（从这里开始）
Content-Type: application/json
Authorization: Bearer eyJhbGciOiJ...
Content-Length: 27
Accept: application/json              ← ② 请求头（到这里结束）
                                      ← ③ 空行（告诉服务器"头部结束了"）
{"name":"test","age":25}              ← ④ 请求体
```

**① 请求行**：三个信息——方法、路径、HTTP 版本
**② 请求头**：一堆键值对，告诉服务器各种元信息
**③ 空行**：就是一个 `\r\n`，分隔头部和体部
**④ 请求体**：实际要发的数据（GET 请求通常没有体）

### 重要请求头（测试开发必须知道的）

| 请求头 | 作用 | 你什么时候会用到 |
|--------|------|-----------------|
| `Host` | 告诉服务器你要访问哪个站点 | 一台服务器可能部署多个网站 |
| `Content-Type` | 请求体的格式 | `application/json` 或 `application/x-www-form-urlencoded` |
| `Authorization` | 身份认证 | 携带 Token 或 API Key |
| `Cookie` | 携带 Cookie | Session 认证 |
| `User-Agent` | 客户端标识 | 服务端可能据此返回不同内容 |
| `Accept` | 期望的响应格式 | 告诉服务器你想要 JSON 还是 HTML |

## 3.3 HTTP 方法：不只是 GET 和 POST

面试官经常问"GET 和 POST 的区别"，但其实 HTTP 有 7 种常用方法。理解它们的核心是理解**两个概念**：

**幂等性（Idempotent）**：同一个请求执行一次和执行多次，结果一样。
- GET /users/123 → 不管执行几次，都是返回同一个用户 → 幂等
- POST /users → 每执行一次就创建一个新用户 → 不幂等

**安全性（Safe）**：请求不会修改服务端资源。

| 方法 | 语义 | 幂等 | 安全 | 场景 |
|------|------|------|------|------|
| **GET** | 获取资源 | ✅ | ✅ | 查询列表、查询详情 |
| **POST** | 创建/提交 | ❌ | ❌ | 创建用户、提交表单 |
| **PUT** | 全量替换 | ✅ | ❌ | 更新用户全部信息 |
| **PATCH** | 部分更新 | ❌ | ❌ | 只更新用户的手机号 |
| **DELETE** | 删除 | ✅ | ❌ | 删除用户 |
| **HEAD** | 同 GET 但不返回体 | ✅ | ✅ | 检查资源是否存在 |
| **OPTIONS** | 查支持的方法 | ✅ | ✅ | CORS 预检请求 |

### GET vs POST 的深层区别（面试标准回答）

```
表面区别（协议规范层面）：
  ① 参数位置：GET 在 URL，POST 在请求体
  ② 长度限制：GET 受 URL 长度限制（浏览器约 2048 字符），POST 理论无限制
  ③ 缓存/书签：GET 可被缓存和收藏，POST 不行
  ④ 幂等性：GET 幂等，POST 不幂等
  ⑤ 语义：GET = 获取，POST = 提交/创建

深层回答（加分）：
  从 TCP 层看，GET 和 POST 没有本质区别，都是 TCP 上的文本数据。
  以上区别都是 HTTP 规范的约定，浏览器和服务器按约定行事。
  理论上 GET 也可以带请求体（但不推荐），POST 也可以用 URL 参数。
```

## 3.4 HTTP 状态码：三档分层记忆法

不要死背所有状态码，用分层记忆法：

```
第一层（必须秒答，测试开发天天见）：
  200  成功
  201  创建成功（POST 成功）
  400  客户端请求有误（参数错误）
  401  未认证（没登录/Token 无效）
  403  无权限（登录了但没权限）
  404  资源不存在
  500  服务器内部错误
  502  网关错误（上游服务挂了）

第二层（面试大概率会问）：
  204  成功但无返回体（DELETE 常用）
  301  永久重定向（URL 永久改变）
  302  临时重定向（URL 临时改变）
  304  资源未修改（用缓存）
  429  请求太多（被限流了）
  503  服务不可用（过载/维护）
  504  网关超时（上游响应太慢）

第三层（知道就行）：
  405  方法不允许
  408  请求超时
  422  参数格式对但语义错
```

**面试高频辨析**（这些对比经常被问）：

| 对比 | 关键区别 | 记忆方法 |
|------|---------|---------|
| 401 vs 403 | 401=没登录，403=登录了但没权限 | 401=门没开，403=门开了但不让进 |
| 301 vs 302 | 301=永久搬家，302=临时出差 | 301 搜索引擎会更新 URL |
| 500 vs 502 | 500=自己挂了，502=上游挂了 | 500=本店故障，502=供货商断货 |
| 502 vs 504 | 502=上游回了错，504=上游没回（超时） | 504 多了个"等不到"的感觉 |

## 3.5 HTTPS 和 TLS：为什么要加密？

HTTP 是明文传输的——你在星巴克用 WiFi 登录一个 HTTP 网站，隔壁桌抓包就能看到你的用户名和密码。

HTTPS = HTTP + TLS 加密层。TLS 解决三个安全问题：

```
① 窃听（Eavesdropping）→ TLS 用加密解决
   数据被加密，即使被截获也无法读取

② 篡改（Tampering）→ TLS 用消息认证码（MAC）解决
   数据被修改能被检测到

③ 冒充（Impersonation）→ TLS 用数字证书解决
   通过 CA 签发的证书确认服务器身份
```

**TLS 握手核心思想**（面试时这样解释）：

> 用非对称加密（RSA/ECC）安全地交换一个对称密钥，
> 之后所有通信都用这个对称密钥加密。
> 
> 为什么这样设计？
> 非对称加密安全但很慢，不适合加密大量数据。
> 对称加密快但密钥怎么安全传输是问题。
> TLS 的巧妙之处就是：用慢的方法安全地协商出快的方法的密钥。

## 3.6 Cookie / Session / Token：三种"记住你是谁"的方式

HTTP 是无状态的——每个请求都是独立的，服务器不记得你之前请求了什么。

那怎么实现"登录态"？三种方案，各有优劣：

```
方案 1: Cookie + Session（传统方案）

  流程：
    登录 → 服务端创建 Session（存在服务端内存/Redis）
         → 返回 Session ID 放在 Cookie 里
    后续 → 每次请求浏览器自动带 Cookie → 服务端用 ID 查 Session

  优点：安全（敏感数据在服务端）
  缺点：有状态（服务端要存），分布式需共享 Session（用 Redis）

方案 2: Token / JWT（现代方案）

  流程：
    登录 → 服务端生成 Token（JWT = Header.Payload.Signature）
         → 返回给客户端
    后续 → 客户端手动放在 Authorization: Bearer <token> 头中

  优点：无状态（服务端不存），天然支持分布式
  缺点：Token 泄露风险，过期管理稍复杂

测试开发场景：
  Cookie/Session → 用 requests.Session() 自动管理
  Token → 先请求登录接口获取 Token，再放到 headers 中
```

### 🔧 动手验证：用 Python 体验两种认证方式

```python
import requests
session = requests.Session()

# 1. 使用 GET 方式设置 Cookie（这是 httpbin 最稳定的设置方式）
# 注意：这里我们手动把 sessionid 和 abc123 作为路径参数
print("正在设置 Cookie...")
session.get('https://httpbin.org/cookies/set/sessionid/abc123')

# 2. 检查 session 对象里是否真的存下了 Cookie
print(f"本地 Session 存储的 Cookies: {session.cookies.get_dict()}")

# 3. 再次请求，看服务器是否识别
print("正在验证服务器端的 Cookies...")
resp = session.get('https://httpbin.org/cookies')
print(f"服务器返回的结果: {resp.json()}")

# === Token 方式 ===
#使用 Python requests 手动模拟：Authorization: 身份验证的标准键名。Bearer : 注意后面有个空格。这是最常见的令牌类型。

import requests

# 1. 模拟你从登录接口拿到的 Token（绝大多数 Token 都是 JWT (JSON Web Token））
my_token = "eyJhbGciOiJIUzI1NiJ9.eyJ1c2VyIjp7IkFsYW4ifX0"

# 2. 手动构造 Headers 字典
custom_headers = {
    "Authorization": f"Bearer {my_token}",
    "Content-Type": "application/json"
}

# 3. 发送请求，手动传入 headers 参数
response = requests.get('https://httpbin.org/headers', headers=custom_headers)

# 4. 验证结果
print(f"状态码: {response.status_code}")
print("服务器收到的 Headers 如下：")
print(response.json()['headers'])
```

# 四、DNS：互联网的"电话簿"

## 4.1 DNS 解决什么问题？

人类记名字（www.baidu.com），机器认数字（14.215.177.39）。DNS 就是那个把名字翻译成数字的系统。

## 4.2 DNS 解析流程

```
你的浏览器要访问 www.example.com

查找顺序（从近到远，找到就停）：
┌─────────────────────────────────────────────────────┐
│ ① 浏览器 DNS 缓存     → 上次查过的还记得？          │ 本机
│ ② 操作系统 DNS 缓存    → 系统级别有缓存吗？          │ 查找
│ ③ /etc/hosts 文件      → 有没有手动配置？            │
└─────────────────────────────────────────────────────┘
          │ 都没找到？
          ▼
┌─────────────────────────────────────────────────────┐
│ ④ 本地 DNS 服务器（运营商提供）                      │ 递归查询
│    → 客户端说"你帮我查到底"                          │ （帮你查到底）
│                                                     │
│    本地 DNS 如果也没缓存，就开始迭代查询：            │
│    ┌───────────────────────────────────────────┐    │
│    │ ⑤ 根域名服务器 (.)                        │    │ 迭代查询
│    │   回复："我不知道，你去问 .com 服务器"      │    │（每级只告诉
│    │                                           │    │ 你下一步
│    │ ⑥ .com 顶级域名服务器                     │    │ 问谁）
│    │   回复："去问 example.com 的权威服务器"     │    │
│    │                                           │    │
│    │ ⑦ example.com 的权威 DNS 服务器            │    │
│    │   回复："www.example.com = 93.184.216.34"  │    │
│    └───────────────────────────────────────────┘    │
│                                                     │
│    本地 DNS 缓存结果 → 返回给你的电脑               │
└─────────────────────────────────────────────────────┘
```

**递归 vs 迭代**：
- 递归查询：你问本地 DNS，本地 DNS 帮你查到底（像打客服电话：你帮我解决）
- 迭代查询：本地 DNS 问根服务器，根说"你去问他"，然后本地 DNS 再去问（像被踢皮球，但自己跑）

## 4.3 DNS 记录类型

| 记录 | 含义 | 面试中的用途 |
|------|------|-------------|
| **A** | 域名 → IPv4 地址 | 最基本的记录 |
| **AAAA** | 域名 → IPv6 地址 | IPv6 版的 A 记录 |
| **CNAME** | 域名 → 另一个域名 | CDN 常用，cdn.example.com → d123.cloudfront.net |
| **MX** | 邮件服务器 | 知道就行 |
| **NS** | 权威 DNS 服务器 | 知道就行 |

### 🔧 动手验证：DNS 查询实操

```bash
# 基础查询
nslookup www.baidu.com
# 看看返回了几个 IP？为什么有多个？（负载均衡）

# 看 ANSWER SECTION 中的记录类型和 TTL

```
