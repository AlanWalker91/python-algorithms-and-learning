# 写在最前面：这份学习路线是怎么设计的

## 先回答一个问题：为什么测试开发要学网络？

你可能会想——"我又不是做运维的，为什么要学网络？"

答案很直接：**测试开发的核心工作场景就是网络。**

你想想日常工作中遇到的问题：
- 接口返回 502 了，是前端的问题还是后端的问题？→ 需要理解 **HTTP 协议**
- 自动化测试偶尔超时失败，是网络抖动还是服务慢？→ 需要理解 **TCP 连接机制**
- 用 requests 调接口，Cookie 怎么就丢了？→ 需要理解 **Session 和 Cookie 的工作原理**
- 压测时连接数打满了，为什么？→ 需要理解 **TCP 状态机和 TIME_WAIT**

这些不是"知道就行"的知识——面试官问你的时候，你需要**讲清楚原因**，而不仅仅是背答案。

所以这份路线的设计原则是：**让你理解"为什么"，而不只是记住"是什么"。**

---

## 学习路线全景图

我把整个学习过程设计成 4 个阶段，每个阶段解决一个核心问题：

```
                     你现在在这里
                         ↓
  ┌──────────────────────────────────────────────────────────────┐
  │                                                              │
  │   阶段 1（1-2周）         阶段 2（2-3周）                     │
  │   ┌──────────────┐       ┌──────────────────┐               │
  │   │ 建立全局观   │──────>│ 深入核心协议     │               │
  │   │              │       │                  │               │
  │   │ 解决的问题： │       │ 解决的问题：     │               │
  │   │ "数据是怎么  │       │ "TCP 为什么可靠？│               │
  │   │  从 A 到 B   │       │  HTTP 请求到底   │               │
  │   │  的？"       │       │  长什么样？"     │               │
  │   └──────────────┘       └────────┬─────────┘               │
  │                                   │                         │
  │   阶段 3（1-2周）                  │  阶段 4（1周）           │
  │   ┌──────────────┐       ┌────────▼─────────┐               │
  │   │ 动手验证     │<──────│ 面试强化         │               │
  │   │              │       │                  │               │
  │   │ 解决的问题： │       │ 解决的问题：     │               │
```

**注意**：阶段 2 和阶段 3 不是严格先后的——我推荐你**交替进行**。学完一个协议就立刻抓包验证，这样记忆效果是纯看书的 10 倍。

## 每个阶段的具体目标

### 阶段 1：建立全局观（地图阶段）

**类比**：你要去一个陌生城市旅游，第一件事是什么？打开地图，搞清楚东南西北、主要道路。不是一上来就钻进小巷子。

**这个阶段你要搞清楚的事情**：
- 网络为什么要分层？分层解决了什么问题？
- 数据从你的电脑到服务器，到底经过了几个"关卡"？
- IP 地址、MAC 地址、端口号——这三个东西各管什么事？

**怎么知道你学会了**：拿一张白纸，画出"浏览器访问百度"的完整数据旅程，全程不看资料、不卡壳。

---

### 阶段 2：深入核心协议（骨架阶段）

**类比**：地图看懂了，现在要深入每个重要景点，了解它的历史和内部结构。

**按重要性排序**（投入时间的比例）：

| 协议 | 面试占比 | 投入建议 | 核心问题 |
|------|----------|----------|----------|
| **TCP** | ~40% | 1 整周 | 它是怎么做到"可靠"的？ |
| **HTTP/HTTPS** | ~30% | 4-5 天 | 一个请求的完整生命周期是什么？ |
| **DNS** | ~10% | 1-2 天 | 域名是怎么变成 IP 的？ |
| **UDP** | ~10% | 1 天 | 它和 TCP 的取舍是什么？ |
| **其他** | ~10% | 穿插 | ARP、NAT、CDN 等 |

**怎么知道你学会了**：面试官随机问一道 TCP 或 HTTP 的问题，你能在 60 秒内给出结构化的回答，并且能回答追问。

---

### 阶段 3：动手验证（肌肉记忆阶段）

**类比**：学游泳不能只看教学视频。你必须下水。

**这个阶段做的事情**：
- 用 Wireshark 亲眼看到三次握手的三个包
- 用 curl -v 看到完整的 HTTP 请求和响应过程
- 用 Python socket 自己写一个 TCP 服务器，理解"监听-接受-收发-关闭"的生命周期
- 用 tcpdump 在服务器上抓包排查问题

**怎么知道你学会了**：给你一段抓包截图，你能在 2 分钟内看出问题所在。

---

### 阶段 4：面试输出（实战阶段）

**类比**：你已经练了很多基本功了，现在要上擂台打实战。核心技能不再是"知道"，而是"说出来"。

**这个阶段做的事情**：
- 对每道高频题，形成自己的"标准回答"（不是背别人的答案）
- 练习"1 分钟讲清一个概念"的能力
- 学会用"总-分-例"的框架组织回答
- 准备 2-3 个"我在实际工作/项目中遇到的网络问题"的故事

**怎么知道你学会了**：找人模拟面试，网络部分拿到"满意"评价。

## 每天怎么学？

我给你一个可操作的节奏，你不需要每天花太多时间，关键是**每天都做**：

```
每天 1.5-2 小时，拆成三个块：

  ┌──────────────────────────────────────┐
  │ 30-40 分钟：理论学习                  │
  │   读对应章节的内容                    │
  │   重点理解"为什么这样设计"            │
  │   画思维导图或写笔记                  │
  ├──────────────────────────────────────┤
  │ 30-40 分钟：动手实践                  │
  │   做章节中的实践任务                  │
  │   写代码、抓包、用命令行工具          │
  │   亲眼验证理论中说的现象              │
  ├──────────────────────────────────────┤
  │ 20-30 分钟：面试题练习                │
  │   从题库中抽 2-3 题                   │
  │   对着手机录音回答                    │
  │   回放检查：有条理吗？卡壳了吗？      │
  └──────────────────────────────────────┘
```

## 一个重要提醒

学网络最容易踩的坑是**试图一次记住所有东西**。

不要这样。网络知识是**网状的**，不是线性的。你第一遍学 TCP 握手的时候可能 60% 懂了，等你用 Wireshark 抓到真实的握手包时会懂到 80%，等你面试时被追问"为什么不是两次"时会懂到 95%。

所以：**允许自己第一遍不完全理解，但要保证动手验证 + 反复回顾。**
